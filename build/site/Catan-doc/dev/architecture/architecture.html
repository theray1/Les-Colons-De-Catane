Architecture

Introduction

Le but de ce chapitre est de spécifier l’architecture d’une famille de produits, dont le projet Les colons de Catane en fait partie.

Il s’agit ici de choix génériques, qui peuvent s’appliquer à plusieurs projets.

Vous pouvez retrouver dans cette partie, l’ensemble des principes d’architectures que nous allons utiliser. Des librairies utilisés, en passant par les différents nœuds logiques, vous pouvez observer comment le serveur est gérer au niveau de son architecture. == Vue physique

La vue physique a pour but de décrire comment devrait être déployé le logiciel selon des nœuds logiques. Ainsi que de définir comment ou plutôt par quel moyen les différents nœuds vont communiquer entre eux. Et enfin le dernier but de cette vue est de donner les différentes contraintes de déploiement, comme quelles librairies sont utilisées, quels sont les besoins matériels.

Au niveau de la vue physique, on retrouve donc 2 nœuds logiques qui sont :

  • Game Server

  • Game Client

Ainsi ces 2 nœuds vont devoir comuniquer et différents protocols vont être utilisé. Ces 2 nœuds ne peuvent pas être réduit, on retrouvera forcément un serveur qui gère les parties, évite qu’il y ait de la fraude et des clients qui font office de passerelles entre le joueur (Personne physique) et le jeu (logiciel). On note pour le serveur, l’utilisation de Java Spring pour la gestion des sockets ainsi que le protocole HTTP pour la première connexion du client. Le noeud logique Game Client utilise TypeScript ainsi que le framework Angular pour gérer l’affichage au sein d’un navigateur web.

Les dépendances principales sont :

  • zone.js

Vue de la fiabilité

Dans cette partie, vous allez présenter les différents choix architecturaux pour assurer la fiabilité du système.

Vous devez aussi présenter les prévisions de fonctionnement dans des conditions limite : * Comment le système est initialisé ? * Comment le système est arrêté ? * Comment sont gérés les failles et le redémarrage du système ?

Utilisez des diagrammes d’activité UML pour décrire l’initialisation et l’arrêt du système.

Prévisions de fonctionnement

INITIALISATION DU SYSTEME :

Un client exécute le logiciel. Si le serveur est éteint, il s’allume.

ARRET DU SYSTEME :

Une fois que toutes les parties se déroulant sur le serveur sont terminées et qu’il n’y a plus de clients en attente de parties, le serveur s’arrête jusqu’à la prochaine exécution du logiciel par un client.

GESTION DES FAILLES ET DU REDEMARRAGE DU SYSTEME :

En cas d’arrêt non souhaité du serveur, par exemple avec une panne du serveur, le serveur est relancé, les parties sont détruites et les clients de la partie encore connectés sont remis dans la file d’attente des participants.
En cas d’arrêt inattendu d’un client jouant à une partie en cours, cette partie se termine immédiatement puis elle est détruite et les clients de la partie encore connectés sont remis dans la file d’attente des participants. Le client qui s’est arrêté devra ré-exécuter le logiciel pour se connecter de nouveau au serveur.
Si c’est le système entier qui s’arrête inopinément, un client devra l’exécuter à nouveau pour relancer le système.

Conventions de codage

Ce projet respecte les conventions JAVA distribuées par Oracle. Cela nous permet d’affirmer que notre code est "propre" selon les normes java présentes à cette adresse : https://www.oracle.com/java/technologies/javase/codeconventions-namingconventions.html.

Règles de traduction des composants

Du côté serveur, on se retrouver avec 2 composants dont un qui ne sera pas codé qui est JavaSpring. Le composant principal que nous nommons Server est l’ensemble des GameManager, GameControler, GameFacade…​

Du côté client, on va également se retrouver avec 2 composants dont un qui ne sera aussi pas codé et qui gère les webSockets, il n’a pas de nom. On va devoir également coder le composant principal que nous nommons, incroyable nom, Client qui regroupe l’ensemble des GameManager, GameControler, GameFacade…​

Règles de traduction des classes

Nous allons interpreter les diagrames UML tout simplement avec un package donne un package, une classe donne une classe…​ Chaque classe utilisera une référence à une interface et non une classe pour minimiser le couplage.

Ainsi, on va se retrouver avec un manager, ce dernier permet coté serveur de gérer l’ensemble des parties et de faire le lien entre ces dernières et Java Spring, et du coté client de faire le lien entre la librairie de websocket et la partie. Pour la partie, c’est le contrôleur qui va gérer cela, faire le lien entre le NetworkControler, la GameFacade et la CommandFactory.

  • NetworkControler : Le contrôleur qui s’occupe de la partie réseau, avec ce dernier, aucune commande ni ressource du GameControler ne sort.

  • GameFacade : Du design pattern Facade, cette classe permet de faire le lien entre le GameControler et le package du jeu. Cela permet de minimiser le couplage.

  • CommandFactory : Du designPattern Factory, cette classe permet de fabriquer les Commands (cf Conception).
